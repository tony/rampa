# k6 — structural analysis

**Classification:** Go · goroutine concurrency with per-VU JavaScript runtimes · mixed closed-loop
and open-loop executors · code-first JavaScript scenarios · single-process local runner.
Pinned at [`grafana/k6@v2.0.0`](https://github.com/grafana/k6/tree/v2.0.0).
License posture: k6 is [`AGPL-3.0`](https://github.com/grafana/k6/blob/v2.0.0/LICENSE.md); this
note summarizes architecture from public source links and is not a source-copying or implementation
plan.

## Execution engine

The scheduler builds executor instances from the script's execution options, reserves planned VUs,
and coordinates setup, scenario execution, teardown, outputs, and thresholds. A VU is an initialized
JavaScript runtime that can be activated by an executor; executors acquire VU handles and drive
`RunOnce` calls. This keeps user JavaScript in the Go process but outside a single shared runtime:
each active VU owns its own runtime state.
→ [`internal/execution/scheduler.go`](https://github.com/grafana/k6/blob/v2.0.0/internal/execution/scheduler.go),
[`internal/js/runner.go`](https://github.com/grafana/k6/blob/v2.0.0/internal/js/runner.go),
[`lib/executor/vu_handle.go`](https://github.com/grafana/k6/blob/v2.0.0/lib/executor/vu_handle.go)

## Scheduling & pacing

k6 makes the scheduling model an explicit executor choice. `constant-vus`, `ramping-vus`,
`shared-iterations`, and `per-vu-iterations` are closed-loop shapes: a VU runs an iteration and then
becomes available for the next one. `constant-arrival-rate` and `ramping-arrival-rate` are open-loop
shapes: iteration starts are scheduled by arrival rate, using preallocated/max VUs as capacity and
recording dropped iterations when no VU is available. That is a real arrival model, not just a
throttle over closed-loop users.
→ [`lib/executor/constant_vus.go`](https://github.com/grafana/k6/blob/v2.0.0/lib/executor/constant_vus.go),
[`lib/executor/shared_iterations.go`](https://github.com/grafana/k6/blob/v2.0.0/lib/executor/shared_iterations.go),
[`lib/executor/constant_arrival_rate.go`](https://github.com/grafana/k6/blob/v2.0.0/lib/executor/constant_arrival_rate.go),
[`lib/executor/ramping_arrival_rate.go`](https://github.com/grafana/k6/blob/v2.0.0/lib/executor/ramping_arrival_rate.go)

## Request/result data structure

Protocol modules emit metric samples through the active VU rather than returning a single global
"result" object. A `metrics.Sample` is a timestamped value tied to a metric and tag set, with optional
high-cardinality metadata; a `SampleContainer` batches samples that share request context. The HTTP
module converts request/response timing, status, and error state into those sample containers.
→ [`metrics/sample.go`](https://github.com/grafana/k6/blob/v2.0.0/metrics/sample.go),
[`js/modules/k6/http/request.go`](https://github.com/grafana/k6/blob/v2.0.0/js/modules/k6/http/request.go),
[`internal/js/runner.go`](https://github.com/grafana/k6/blob/v2.0.0/internal/js/runner.go)

## Metric & percentile data structure

Metrics are typed (`Counter`, `Gauge`, `Rate`, `Trend`) and accumulated into matching sinks.
Counters and rates are exact scalar accumulators. Local `TrendSink` keeps a slice of float values,
sorts lazily on percentile query, and reports min/max/avg/median/p90/p95 for summaries and
thresholds. That makes local percentiles exact for retained samples but not a bounded, mergeable
distributed summary in the way locust histograms or artillery DDSketch periods are.
→ [`metrics/sink.go`](https://github.com/grafana/k6/blob/v2.0.0/metrics/sink.go),
[`metrics/thresholds.go`](https://github.com/grafana/k6/blob/v2.0.0/metrics/thresholds.go)

## Distributed / aggregation

The open-source local runner is a single Go process. Outputs receive batches of
`metrics.SampleContainer`s and are expected to flush asynchronously; cloud/output adapters can turn
samples into their own buffered or bucketed wire formats, but that is not a local master/worker
coordination protocol like locust or jmeter. Structurally, local k6 scales by one process first and
uses external orchestration or a service boundary for fleet execution.
→ [`output/types.go`](https://github.com/grafana/k6/blob/v2.0.0/output/types.go),
[`output/helpers.go`](https://github.com/grafana/k6/blob/v2.0.0/output/helpers.go),
[`output/cloud/expv2/collect.go`](https://github.com/grafana/k6/blob/v2.0.0/output/cloud/expv2/collect.go)

## Scenario / user API

Code-first JavaScript defines lifecycle functions and scenario options: `setup`, scenario functions
including `default`, `teardown`, thresholds, and executor configurations. The runtime also exposes
`SharedArray` for init-time data sharing without copying large fixtures into every VU runtime. This
is closer to locust's code-first user model than to jmeter's XML tree, but with executor scheduling
declared separately from request code.
→ [`lib/executors.go`](https://github.com/grafana/k6/blob/v2.0.0/lib/executors.go),
[`internal/js/runner.go`](https://github.com/grafana/k6/blob/v2.0.0/internal/js/runner.go),
[`internal/js/modules/k6/data/data.go`](https://github.com/grafana/k6/blob/v2.0.0/internal/js/modules/k6/data/data.go)
